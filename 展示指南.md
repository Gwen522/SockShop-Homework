## 一、先看结论：现场到底要不要跑代码？

**大概率不用现场跑。** 5～6 分钟要讲完四个阶段，平均每阶段一分多钟，根本没时间等程序跑（Selenium 跑一遍要 40 秒，JMeter 压测也要 1 分钟）。

**推荐做法：直接讲报告里的结果 + 给现成的截图/图表。** 又快又稳，不会现场翻车。

但如果评委/老师**当场点名要看运行**，我们也准备好了，照下面「二」操作即可。

## 二、万一要现场跑，怎么操作（傻瓜版）

### 准备工作（上台前先做好，别等台上才弄）
1. 确认 Docker Desktop 已经打开（菜单栏有只小鲸鱼图标，不在转圈）。
2. 打开「终端」，复制粘贴这一行回车，确认网站活着：
   ```
   curl -s -o /dev/null -w "%{http_code}\n" http://localhost:8079/
   ```
   显示 `200` 或 `301` 就说明 SockShop 在跑，没问题。

### 现场演示（三选一，全都是一个命令搞定）

在「终端」里先进入项目目录（把下面路径换成你电脑上项目所在的位置）：
```
cd 你存放项目的路径/sockshop-testing
```

然后运行这一个脚本，会弹出菜单让你选：
```
zsh demo.sh
```

菜单长这样，**输入数字 + 回车**即可：

| 输入 | 演示内容 | 大家会看到什么 | 耗时 |
|------|----------|----------------|------|
| **1** | Selenium 功能测试 | **浏览器自动弹出**，自己浏览商品→注册→加购→看购物车，全程自动操作 | 约 30 秒 |
| **2** | JMeter 性能测试 | 终端实时打印压测进度，最后显示成功数、错误率 | 约 1 分钟 |
| **3** | 直接看结果（最稳） | 自动打开 JMeter 网页报告 + 性能图表 | 秒开 |

**台上最推荐按 `3`**——秒开，直接展示图表讲解，不用干等。
**想秀一把「自动化」效果就按 `1`**——浏览器自己动起来，视觉冲击强，但要多花 30 秒。

> 小贴士：演示选 1 时，如果 Chrome 那台电脑有问题（之前出过代理的毛病），可以打开
> `selenium/demo_visible.py`，按文件里第 20 行附近的注释把浏览器换成 Safari。

## 五、可能被问到的问题（有个心理准备）

- **Q：吞吐量为什么 200、300 并发反而降了？**
  A：主要是那两组每个用户循环次数少、测试时间窗口短，统计出来的吞吐偏低，不是系统扛不住——
  错误率一直是 0，响应时间反而更快。这点报告分析里也写了。

- **Q：Selenium 那个操作时间为什么要 3 秒？**
  A：那不是服务器慢，是我们脚本里特意加了等待，怕页面没加载完就去判断会出错。真实服务器响应
  速度看 JMeter 的数据，都在 10 毫秒以内。

- **Q：为什么用 Safari 也测？**
  A：作业要求验证跨浏览器兼容性，所以 Chrome 和 Safari 都跑了一遍。
